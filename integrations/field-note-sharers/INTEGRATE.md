# field-note-sharers

The canonical attribution-layer pattern for a Docusaurus 3 reference wiki. A flat section folder, one Markdown file per source, where each source is either a named individual (mentor, practitioner, author) or a firm (consultancy, lab, VC) whose work the wiki credits.

The noun: a **field-note-sharer** is one source. The section folder holds many of them. The integration packages the canonical shape so a new wiki gets it for free and existing wikis stay coherent across the family.

## What it does

Adds (or formalizes) a section to the wiki for crediting external sources. Two things appear on the public site:

- **The section index** (`/<section>/`) listing every source the wiki cites, briefly, with a one-line stance for each.
- **One page per source** (`/<section>/<source-slug>`) holding the source's profile, why they are on this wiki, what they have contributed, and (optionally) a chronological log of field notes captured from specific pieces of their published work.

The section is reachable from the sidebar via an autogenerated category, so new source pages appear automatically on next build.

## Why it exists

Most reference wikis end up needing an attribution layer. Without one, the wiki's claims float without provenance, and over time the prose drifts toward "AI consultancy boilerplate." The field-note-sharer pattern fixes both problems with a single sub-folder convention.

The wikis in this family that have already shipped some version of this pattern:

- `agenticbusiness.wiki` uses `/mentors/` (named individuals + private-conversation archetypes who contribute principles).
- `appliedai.wiki` uses `/note-sharers/` (firms whose public publications get processed into dated field notes) and `/people-to-follow/` (individual practitioners worth tracking).
- `truthmanagement.wiki`, `hyperagency.wiki`, `supersuit.wiki` do not yet ship an attribution layer; this recipe is what they would adopt.

The canonical shape is intentionally flat: **one file per source**, not one folder per source. Multiple field notes about the same source accumulate as dated sub-sections inside that one file. The folder-per-source variant (one folder per firm, dated files inside) exists on `appliedai-wiki/docs/note-sharers/` as a richer evolution, and is out of scope for this recipe — that shape is for wikis with high-volume per-source publishing throughput. Most wikis only need the flat shape.

## Configurability

The integration is configurable in three places. The user picks these at install time (or the generator prompts for them on a fresh wiki).

### 1. Section folder name

Pick the name that matches the wiki's register:

| Name | Connotation | Used by |
|---|---|---|
| `mentors` | Named individuals who shape the wiki's worldview; principal-audience wikis | agenticbusiness.wiki |
| `people-to-follow` | Practitioners worth tracking; implementer-audience wikis | appliedai.wiki |
| `note-sharers` | Firms whose publications you process into field notes | appliedai.wiki (separate folder-per-firm shape) |
| `sources` | Generic; works when the section mixes individuals and firms | (none yet) |
| `field-note-sharers` | The literal noun; works when the wiki is about attribution itself | (none yet) |

Other names are fine — the recipe just installs whatever the user picks.

### 2. Whether to include private-conversation archetypes

Some wikis (e.g., agenticbusiness.wiki) credit operators who shared principles in private conversation, anonymized as archetypes ("The Reformed Wall Street Operator"). The section index documents the source category explicitly.

If the wiki only cites public sources, drop the archetype scaffold from the index template.

### 3. Whether to include dated field notes inside each source file

The simplest shape: each source file is just a profile + contributions + further reading. The Priestley-style shape: each source file additionally has a `## Field Notes` section that grows over time, with dated `### YYYY-MM-DD — <piece title>` sub-sections capturing specific lifts from specific videos, books, articles, or talks.

Default: include the `## Field Notes` section scaffold (commented out if empty). Wikis that don't want a dated log can delete the section from the template before populating.

## Files

```
INTEGRATE.md                       (this file)
templates/
  section-index.md.template        → wiki/docs/<section>/index.md
  source-page.md.template          → wiki/docs/<section>/<source-slug>.md
```

Both templates use simple `{{PLACEHOLDER}}` substitution. The user fills these in by hand on a small wiki, or via the generator on a fresh wiki.

## Install steps

1. **Pick the section folder name.** See the table above. For this walkthrough, assume `mentors`.

2. **Create the section folder.**

   ```bash
   mkdir -p wiki/docs/mentors
   ```

3. **Copy the section index template.**

   ```bash
   cp curated-wiki-integrations/integrations/field-note-sharers/templates/section-index.md.template \
      wiki/docs/mentors/index.md
   ```

   Open the file and replace every `{{PLACEHOLDER}}` (see the template for the list — section name, audience, source categories you actually use).

4. **Copy the source-page template, one per source.**

   ```bash
   cp curated-wiki-integrations/integrations/field-note-sharers/templates/source-page.md.template \
      wiki/docs/mentors/daniel-priestley.md
   ```

   Replace placeholders. Either keep or delete the `## Field Notes` section depending on whether the wiki tracks a chronological log per source.

5. **Wire the sidebar** in `wiki/sidebars.ts`. Add a category that autogenerates from the section folder:

   ```ts
   {
     type: 'category',
     label: 'Mentors',                          // human label for the section
     collapsed: false,                           // or true if collapsed-by-default
     link: { type: 'doc', id: 'mentors/index' }, // section landing
     items: [
       { type: 'autogenerated', dirName: 'mentors' },
     ],
   },
   ```

   New source files in `docs/mentors/` will appear automatically on next build. The autogenerated sidebar items generator in `docusaurus.config.ts` (`sidebarItemsGenerator` — ships in `wiki-template/`) auto-excludes `index.md` from being listed as a duplicate child.

6. **Build to verify.**

   ```bash
   npm run build
   ```

   `onBrokenLinks: 'throw'` will fail the build if a cross-link is dangling. Fix and rebuild before committing.

## Page anatomy

Each source page follows the family's canonical page anatomy:

```markdown
---
title: "Source Name"
slug: /<section>/<source-slug>
description: "One sentence under 200 chars naming who they are and what they are on this wiki for."
---

# Source Name

*Italic one-line definition. Quotable as the subject or object of a sentence.*

---

## Background

Where they have operated, what they have built, what gives them standing.

## Why they are on this wiki

What specifically this source contributes that the wiki could not assemble otherwise.

## Contributions

The principles, concepts, perspectives, or playbooks the source has shaped on this wiki.
List with absolute-path cross-links to those pages.

## Field Notes

Chronological log of specific lifts from this source. Optional. Delete the section
if the wiki does not maintain per-source logs.

### YYYY-MM-DD — "Piece Title" (Format)

What was lifted. Verbatim quotes in blockquotes. Where each insight landed on the wiki.

## How to cite them

Citation format for the wiki's authors. For published-book or public-teaching sources, the format is the public name + source title + date.

## Further Reading

- [Related page](/section/slug)
- [Section index](/<section>)
```

## Gotchas

- **Flat structure only.** This recipe is the one-file-per-source shape. If the wiki needs the folder-per-source-with-dated-files shape, that is a separate (richer) evolution living on `appliedai-wiki/docs/note-sharers/` — copy that pattern directly rather than extending this recipe.
- **Section folder name has to match the `dirName` in the sidebar.** Renaming the folder later requires updating `sidebars.ts` and every internal cross-link to that section.
- **Italic def line is required on every page** in this section, same as anywhere else in the wiki. It is the page's quotable spine.
- **The section index is mandatory** even if there is only one source page in the section at first. The index documents the source categories and the inclusion bar, both of which compound in value as the section grows.
- **Don't redefine concepts inline on a source page.** If a source contributed a concept that has its own canonical page elsewhere, cross-link to that page from the contributions list. Source pages explain who and why, not the substance of each contribution.

## Alternatives considered

- **Folder-per-source with dated files inside** (the `note-sharers/` shape on appliedai.wiki). Richer but heavier. Use when a single source publishes frequently enough to merit a sub-folder of dated entries; otherwise the flat shape is cleaner.
- **Frontmatter-tag-based attribution** (each principle page has a `source:` field, no dedicated section). Simpler but loses the index-and-profile surface, which is where readers actually browse "who does this wiki trust." Rejected.
- **External JSON / YAML data file** rendered into a single attributions page. Brittle to source-specific prose. Rejected.

## Status

Ready. Used in production on `agenticbusiness.wiki/mentors/` and `appliedai.wiki/people-to-follow/`.
